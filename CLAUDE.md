# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 작업 워크플로우 (필수 사이클)

**모든 작업은 아래 6단계 사이클을 따른다. 단계 생략 금지.**

```
1. 계획 작성  →  2. 코드 구현  →  3. 1차 검수/리뷰 (codex_mcp 크로스리뷰, 합의 도달까지)
                                          ↓
                                       3a. 수정
                                          ↓
                                       4. 2차 검수
                                          ↓
                                       4a. 수정
                                          ↓
                                       5. 완료 보고서를 계획 문서에 추가
```

### 작업 규모별 적용
- **간단한 작업**: 한 번에 위 사이클 1회 실행
- **큰 작업**: 작업을 독립 단위로 쪼개고, **쪼갠 각 범위마다** 위 사이클을 1회씩 수행
  - 큰 작업 시작 전, "쪼개기 계획"을 먼저 DECISIONS/에 기록
  - 각 sub-task는 자체 계획 문서를 갖고 그 안에 완료 보고서를 누적

### 1. 계획 (Plan)
- Plan Mode 또는 별도 계획 문서로 구현 계획 수립
- 다관점 분석: 성능/유지보수성/확장성/단순성 등 여러 관점에서 방법 구상 후 최적안 도출
- `mcp__codex__codex`(GPT 전문가) 크로스 피드백: Architect/Plan Reviewer/Scope Analyst 등 적절한 전문가 호출
- 사용자 승인 후 구현 시작

### 2. 구현 (Implement)
- 메인 세션은 계획 검토와 진행 관리에 집중, 실제 코드 수정은 서브에이전트에 위임
- 독립적인 작업 단위는 **가능한 한 여러 서브에이전트를 병렬로 소환**
- 서브에이전트 소환 시 반드시 포함:
  - 수정 대상 파일 경로와 변경 내용 명세
  - 관련 컨텍스트 (기존 코드 패턴, 재사용 가능한 함수/유틸)
  - 변경 범위 경계 (무엇을 하지 말아야 하는지)
- 서브에이전트 결과를 메인 세션에서 검토하고 파일 간 정합성 확인
- 계획 이탈이 필요하면 사용자에게 먼저 확인

### 3. 1차 검수/리뷰 (Cross-Review with Codex MCP)
- 구현이 끝나면 `mcp__codex__codex` Code Reviewer / Security Analyst 등 적절한 전문가에게 리뷰 위임
- 메인 세션도 독립적으로 코드 리뷰
- 양측 의견이 **합의에 도달할 때까지** 토론 (이슈, 합의/불합의, 근거 명시)
- 합의된 수정 사항을 반영

### 4. 2차 검수
- 수정된 결과를 다시 검수 (테스트 통과, side effect 없음, 계획 이탈 없음)
- 필요 시 추가 수정
- 모든 검증 명령 통과 확인 (`ruff/mypy/pytest`)

### 5. 완료 보고서
- 계획 문서 끝에 **완료 보고서 섹션**을 추가
- 포함 내용:
  - 실제 변경된 파일/라인
  - 추가/수정된 테스트
  - 검수에서 잡힌 이슈와 해결
  - 검증 결과 (테스트 결과, 사이드이펙트 확인)
  - 잔여 follow-up 또는 후속 작업

### 6. 에러 이력 관리 (ERR_HIST/)
- 테스트 중 같은 종류의 에러가 **2회 이상** 발생하면 `ERR_HIST/`에 기록
- 파일명 형식: `YYYY-MM-DD_HHmm_{에러_요약}.md`
- 내용: 에러 내용, 원인 분석, 해결 방법, 재발 방지 포인트
- 구현 전 `ERR_HIST/` 참조하여 동일 실수 방지

## 의사결정 기록 (DECISIONS/)

다관점 분석 후 내린 기술적 결정은 `DECISIONS/` 폴더에 기록한다.

- 파일명 형식: `YYYY-MM-DD_{결정_주제}.md`
- 내용: 배경, 검토한 선택지들, 선택 이유, 트레이드오프
- 이미 기록된 결정은 재논의하지 않는다 (변경 필요 시 새 결정으로 기록)

## 컨텍스트 인수인계

복잡한 작업이 세션을 넘어갈 때, 세션 종료 전 현재 상태를 기록한다.

- 기록 위치: `HANDOFF.md` (프로젝트 루트)
- 내용: 현재 진행 상태, 완료 항목, 남은 작업, 주의사항
- 새 세션 시작 시 `HANDOFF.md`가 있으면 먼저 읽고 이어서 작업
- 작업 완료 후 `HANDOFF.md` 삭제
