## 확정된 설계 결정 (사용자 논의 2라운드 — 재논의 불가)

1. **F4-c'' = LLM 스키마 기반 프로파일 참조 선택** (사용자 제안 채택):
   - 키워드 사전 감지(1차 제안)는 주 경로로 기각 — 패러프레이즈에 조악.
   - LLM 프롬프트에 유저의 **실제 값이 아니라 스키마만**(닫힌 클래스 enum +
     설명 + 예시) 제공 → 기존 질의해석 호출 1회에 `profile_refs` 선택 출력 추가
     (추가 호출/비용 0) → enum 검증 게이트(밖이면 드롭) → 시스템이 로그인
     유저의 해당 필드 개념을 **결정적으로 조인**해 기존 질의 주입 경로로 부스트.
   - 개인화 에이전트 field_router("query/intent → field groups", 키워드 맵,
     상담 컨텍스트용)와 **모양 유사·소비처 상이** — 클래스 정의를 그쪽
     FIELD_GROUPS와 정렬(참조만, 런타임 결합 없음), 추후 서비스 노출 시 교체
     가능한 얇은 지점으로 유지.
2. **F5 = 전체 그래프 뷰가 기본**: 상품 1개 중심 에고 뷰는 부차. 유저+상품+
   개념 혼합 전체 그래프 → 노드 선택 시 그 중심 포커스. (사용자: "찐")
3. **데이터 정리 확정**(혼동 해소, 09에도 반영): 상품 카탈로그 1종(517).
   리뷰는 2층 — 원천 통계(상품당 수십~수백, 사이드카 숫자)와 **NLP 트리플
   906건(그래프 재료, 517상품 전부 보유하나 70%는 1건)**. dense_golden은 별도
   상품이 아니라 906건을 32상품에 몰아둔 **테스트 전용 합성 픽스처**. 그래프
   밀도의 근본 해법 = 리뷰 백엔드(실 rs.jsonl 적재) — "dense로 시연" 제안 철회.

## F4 배치 — 질의 반영 정합 (a + c'')

### F4-a. 반영도 표시 일관화 (S)
- LLM 경로도 미반영 표현을 항상 표면화: 프롬프트의 unresolved 지시 강화
  ("의미 있는 표현 중 개념으로 확정하지 못한 것은 unresolved_terms로 반환" —
  현 프롬프트 확인 후 부족분 보강) + 응답 게이트에서 unresolved 통과 확인.
- 결과 상단 **반영 요약 1줄**: "이 결과: [해석된 개념들] + 내 프로필 반영
  [클래스들] 기반" — 프론트는 기존 해석 영역에 요약줄 렌더 추가(칩/배너 재사용).

### F4-c''. 프로파일 참조 선택·주입 (M)
- **클래스 enum**(모듈 상수, serving user profile 필드와 1:1 매핑 표 동봉):

  | class | 프로파일 필드 | 예시 트리거(프롬프트 예시용) |
  |---|---|---|
  | `concerns` | concern_ids | "내 고민", "피부 고민에 맞는" |
  | `skin` | (1차: concern_ids로 대리 — basic 스킨타입은 개념 id가 아님, 스킨타입→concern 사전 매핑은 후속) | "피부타입", "내 피부에 맞는" |
  | `goals` | goal_ids | "내 목표 효능" |
  | `preferred_brands` | preferred_brand_ids | "좋아하는 브랜드" |
  | `preferred_keywords` | preferred_keyword_ids | "내 취향 질감" |
  | `repurchase` | repurchase_brand_ids/repurchase_category_ids | "자주 사는", "재구매하던" |
  | `owned` | owned_product_ids | "내가 산", "보유 제품이랑 어울리는" |

- **LLM 확장(코덱스 #3)**: 프로바이더 JSON Schema는 없음(프롬프트 예시가 계약,
  Azure json_object) — 프롬프트 JSON 예시에 `profile_refs: []`와
  **`unresolved_terms: []`(F4-a 필수 — 현재 출력 스키마에 없음)** 둘 다 추가.
  **게이트는 기존 검증이 있는 `_interpret_with_llm`에**: profile_refs는 enum
  멤버만+중복 제거+**≤3 상한**, unresolved_terms는 개수(≤5)/길이 제한 후 기존
  unresolved와 병합. 비로그인이면 선택돼도 주입 스킵(클래스 표시만).
- **주입 어댑터(코덱스 #1 — `_inject_query_preferences` 재사용 아님)**:
  `_inject_query_preferences`는 resolved_concepts+단일 scope 전제라 profile
  refs를 거기 합치면 **후보 축소(narrowing)에도 질의 개념으로 쓰임** — 금지.
  별도 어댑터 `_apply_profile_refs(user_scoped, classes)`: 클래스→edge_type
  매핑(concerns/skin→`HAS_CONCERN`, goals→`WANTS_GOAL`, preferred_brands→
  `PREFERS_BRAND`, preferred_keywords→`PREFERS_KEYWORD`, repurchase→
  `PREFERS_CATEGORY`/`PREFERS_BRAND`), **각 항목의 원 scope 보존**,
  resolved_concepts 비합류(축소 무영향).
- **중복 의미론(코덱스 #2)**: `(edge_type, normalized_id, scope_group)` 기준
  idempotent dedup — 이미 활성인 동일 선호는 **추가 가중 없음**(set 수집이라
  점수상 no-op임을 명시). `query_injected_ids`와 `profile_ref_ids`를 **분리
  추적**해 후처리 라벨링이 저장 선호를 "질의 언급"으로 오표시하지 않게 함.
- **응답 페이로드(코덱스 #4)**: `interpretation.profile_refs`엔 검증된 클래스명만.
  실제 개념/라벨은 서버가 get_user 조인 후 별도
  `applied_profile_refs: [{class, concepts(labels), injected}]`로 구성 —
  **익명 요청엔 이 필드 자체를 생략**(응답 shape 불변 + 익명 결과 id/점수
  동일성 테스트).
- `owned` 특례 — 이미 G4 boost로 상시 반영 중(코덱스 검증: `_run_scored_pipeline`
  공용 경로): **주입 금지, "반영 중" 표시만**.
- **칩 표시**: 해석 영역 "내 프로필 반영: 고민(진정·보습) · 재구매(이니스프리)"
  — applied_profile_refs 기반(F4-a 요약줄과 한 세트).
- **폴백(LLM off)**: 동일 enum에 대한 최소 보수 사전 트리거(클래스당 2~3
  표현) — degraded 모드로 동작, LLM 대비 재현율 낮음을 docstring 명시.
- **테스트**: enum 게이트(밖 드롭)·클래스별 주입 정합(스코프 포함)·owned
  비주입 특례·비로그인 스킵·LLM off 폴백·칩 payload·기존 ask/search 무파괴
  (스냅샷 diff 0 — 질의 없는 /api/recommend 경로 무접촉).

## F5 배치 — 전체 그래프 뷰 + 포커스 (M~L)

- **규모(코덱스 실측 확정)**: wide 전체 canonical **약 1,110노드 /
  5.7~6.3k 엣지**(승격 top_* 고유 개념 35 + truth/user 개념). 구현 시 재확인만.
  실카탈로그 45k 전환 시엔 cap 선적용 필수(문서화).
- **노드 정체성 원칙(코덱스 #5 — 이것 없으면 유저/상품이 두 섬)**: 개념당
  **canonical 노드 id 하나**(추천 조인과 동일 정규화 — 브랜드/goal/concern의
  상품측·유저측 id 차이 해소), scope·edge 의미는 **엣지 메타데이터**로(기존
  user graph의 `id|scope:*` 노드 방식 미사용). 유저-개념은
  `scoped_preference_ids`, 유저-상품 owned는 `extract_owned_product_ids`로
  **실상품 노드에 연결**. `SHARES_ATTRIBUTE`는 unordered pair dedup.
- **API `GET /api/graphs/full`**: 노드 = 유저(가명)+상품+승격 개념(타입 라벨),
  엣지 = 상품-개념(승격 신호), 유저-개념(선호/고민), 유저-상품(owned),
  상품-상품(SHARES_ATTRIBUTE, 기존 캡). 쿼리 파라미터: `edge_types`(토글),
  `min_strength`(약신호 숨김), `max_nodes`(가드). 응답은 기존 그래프 계약
  (nodes/edges) 재사용.
- **프론트(graph_view.js/app.js)**: 그래프 뷰어 기본 = 전체 뷰. 타입 범례+
  토글(유저/상품/개념·엣지 타입별). **포커스 인터랙션**: 노드 클릭 → 이웃
  하이라이트+비이웃 흐림+줌, 더블클릭 → 그 노드 중심 에고 재레이아웃(기존
  상품별 뷰가 이 안에 흡수 — 상품/유저 모두 지원). 레이아웃 fcose/cose 초기
  1회, 재레이아웃은 포커스 시에만.
- **레이아웃/성능(코덱스 #6)**: fcose는 **미설치 — 계획에서 제거**, 내장
  `cose` 유지. 전체 모드에선 **엣지 라벨 숨김**(포커스/줌인 시에만 표시), 노드
  라벨은 타입별 축약. **숫자형 완료 기준**: wide(≈1.1k 노드) 초기 레이아웃+
  렌더 **≤5초**(브라우저 실측), 포커스 반응 ≤0.5초, 기존 상품별 뷰 무회귀.
- **가드(코덱스 #7)**: `min_strength`는 스케일이 다른 타입에 일괄 적용 금지 —
  **score-bearing 엣지(SHARES_ATTRIBUTE 등)에만** 적용하거나 타입별 정규화.
  `max_nodes`는 **결정적 절단 우선순위**(유저·상품 우선, 개념은 연결도순) +
  살아남은 endpoint 기준 엣지 필터(dangling 금지) + 응답 메타
  `truncated/total_nodes/total_edges`.
- **테스트**: full 엔드포인트 계약(타입·토글·가드), 가명 유저 라벨(실데이터
  원문 미노출 — 가명 id만), 서버 계약 테스트 + 브라우저 시각 검증(메인 세션).
