## 발단

사용자: 리뷰에서 타제품을 정확히 찾는 상품↔상품 직접 엣지는 어렵다 치자. 그런데
상품에 붙은 속성 노드(보습·촉촉·성분·브랜드…)를 공유 노드로 삼으면 그걸 통해
상품이 2홉/멀티홉 연결되지 않나? A의 '촉촉'과 B의 '촉촉'은 같은 노드니까. Relation
프로젝트 new_graph_system/project_3에서 그렇게 멀티홉을 만들었는데, GraphRapping은
지금 그게 안 되나?

→ 리서치 3트랙(Relation 패턴 / GraphRapping 기질 실측 / 설계공간)으로 분석 →
"직관 맞음, 데이터 실재, 명시적 기능만 미구현" 결론 → Phase 8 계획 → 4개 결정을
놓고 아래 논의.

## 결정 1 — 유사도 메커니즘 (3단계 정정)

**초기 오해(내가 유발)**: "성분 ∩ 효능 결합"이라 써서, 마치 (a) 두 노드를 합쳐 새
노드를 만들거나 (b) 두 축이 다 맞아야만 유사로 치는 하드-AND처럼 읽혔다.

**사용자 정정**: 셋 다 아니다. 기준 상품에서 공유 노드를 갖는 후보들이 나오고,
각자 공유 노드의 개수·가중치가 다르니 **그 점수로 이미 가장 유사한 상품이 레벨링**
된다. 왜 굳이 축을 하드-AND 하나?

**확정**: 사용자가 옳다. 메커니즘 =
- 노드는 축별로 **구분된 채** 유지 (병합·새 노드 없음)
- 유사도 = **공유 노드들의 IDF 가중 합** (다축이 저절로 반영 — 여러 축 공유 시 고점수)
- 랭킹 + top-N이 유사 상품 자동 선별
- 내가 인용한 "성분∩효능 노이즈 6.1%"는 하드-AND 지시가 아니라 **"다축 공유 쌍일수록
  깨끗하다"는 진단** = 가중 합 방식이 옳다는 근거였음

## 결정 1 후속 — 카테고리 게이팅은 "맥락 파라미터" (핵심)

**사용자 통찰**: 카테고리 게이팅 여부는 유사도 계산의 속성이 아니라 **어떤 추천이냐**에
달렸다.
- **유사상품 추천**(item-to-item): 게이트 **ON** — 메이크업의 유사상품이 헤어제품이면
  말이 안 됨
- **일반 추천**("A유저가 상품1 자주 구매 → 상품1과 연관 많은 다른 제품"): 게이트
  **OFF** — 하드게이트는 추천 다양성을 억제. 카테고리는 점수 축으로만
- **쿼리 기반 추천**(검색/채팅): 쿼리에 카테고리가 언급됐으면 이미 그 풀로 좁혀져
  **상류에서 게이팅**됨 → 유사도 단계에서 또 걸 필요 없음

**확정**: G1(계산)은 **카테고리 비종속**(카테고리 = 점수 노드) + 카테고리 정보 노출.
**게이트는 각 소비 서피스가 `category_gate` 파라미터로 결정.** 유사상품(G2/G3)=ON,
일반 추천 boost(G4)=OFF, 쿼리(G5)=상류 게이팅.

## 축별 재해석 (사용자 도메인 지식 반영)

- **카테고리**: 데이터 풀이 작아 동어반복처럼 느껴질 뿐, 중요한 정보. → 점수 기여
  노드로 유지(하드 게이트는 위 맥락 규칙대로).
- **브랜드**: 노이즈 아님. **지금 데이터가 특정 브랜드에 쏠린 것**(이니스프리 186상품)
  뿐. → 하드 제외/캡 안 씀. **IDF가 쏠림 자동 보정**(대형 브랜드 가중≈0, 니치 라인
  가중 높음).
- **bee_attr & keyword** (데이터 모델 정정):
  - bee_attr = 뷰티 속성 **클래스(분류체계)** — 리뷰 구문을 이 클래스로 분류하는 체계.
    **클래스명 자체를 공유 노드로 쓰면 안 됨** ✓
  - bee_attr를 노드화한 이유 = **하위 keyword 레벨을 정확히 구분하기 위함.**
    "bee_attr는 다른데 keyword 값이 같은" 경우가 동일 노드로 뭉개지면 안 되므로.
  - **keyword 하위 레벨이 사용자가 쓰려는 주요소.** → keyword 공유 노드 키 =
    **(bee_attr_id, keyword_id) 복합키.** 신호가 이미 두 필드를 따로 보유
    (`wrapped_signal`의 `keyword_id` + `bee_attr_id`, provenance_provider.py 확인).
  - **keyword 커버리지가 지금 낮은 것(wide 27)은 데이터 이슈** — bee 데이터의
    keyword 분리/분석이 파이프라인에 덜 반영된 것뿐, **채워질 것.** 설계 문제 아님.

## 결정 요약 (4건 확정)

1. **유사도 = IDF 가중 공유노드 합 + 랭킹 + top-N** (하드-AND·병합·하드게이트 없음).
   카테고리 게이팅은 **소비 맥락 파라미터**(유사상품=ON / 일반추천=OFF / 쿼리=상류).
   축: keyword(bee_attr 복합키, 주력)·ingredient(주력)·category·brand·main_benefit
   전부 점수 노드. bee_attr(클래스명)은 점수 제외·keyword 스코핑 전용. polarity 반영.
   브랜드/카테고리는 IDF가 쏠림 처리(하드 배제 안 함).
2. **첫 활용 = G2 그래프 뷰 먼저** (연결 시각화·"그래프 살아있음" 실증), 이후 G3 위젯.
3. **저장 = ephemeral 우선** (D1/D2 선례, 스키마 표면 0), 영속은 실측/SLA 후.
4. **keyword = 1급 주력 축**(bee_attr 복합키 스코핑), "관찰용" 아님. bee_attr는
   스코핑 전용. 현재 낮은 커버리지는 데이터 성숙으로 회복.

## 완료 기준 변경 (결정 1의 귀결)

가중+랭킹 방식이므로 "전체 쌍의 교차카테고리 비율(노이즈율)"은 더 이상 핵심 지표가
아니다. → **기준 상품별 top-N 유사 상품의 품질**로 판정:
- 게이트 ON 맥락(유사상품): top-N이 동일 카테고리 + 납득 가능한 공유 근거로 채워지나
- 게이트 OFF 맥락(일반 추천): 앵커 상품과 실제로 연관성 높은(공유 노드 多) 상품이 상위인가
- 커버리지: 유사 이웃 ≥1 보유 상품 비율(빈 위젯 방지)

## 추기 — 3자 재리뷰 + keyword 복합키 재확정 (2026-07-15 후속)

Opus·Sonnet·Codex 3자 재리뷰(논의록을 확정제약으로 공유). 3/3이 **동일 사실오류**
수렴: keyword 복합키 `(bee_attr_id, keyword_id)`가 **서빙 프로파일엔 없음** —
`aggregate_product_signals` groupby가 `(product, edge_type, dst_type, dst_id)`라
bee_attr·polarity를 집계 단계에서 버림. 계획이 근거로 든 `provenance_provider`는
raw `wrapped_signal`을 읽어(다른 소스) 참이었고, attach가 실제 읽는 서빙엔 재료 없음.

**사용자 재확정(중요)**: 복합키 병합(Option C)은 안 된다. "가볍다"가 제형/발림성/
패키지에서 **다른 의미**이므로 bee_attr로 구분하는 게 설계 의도이자 주류 케이스다.
BEE 분류 정확도는 신뢰하고 가며, "보습"이 7개 bee_attr에 퍼진 건 BEE 추출 스팬
정밀도의 사이드이펙트일 뿐 — 시스템을 모델 부정확 전제로 설계하지 않는다. 즉
**결정 4(복합키+극성) 유지.**

**해소(사실제약)**: keyword 축만 **raw `wrapped_signal` sidecar**에서 소싱 —
DB는 `idx_ws_product` 인덱스로 per-product 조회(컬럼 bee_attr_id·keyword_id·polarity
실재 확인), demo는 `product_signals`. 집계/서빙 keyword 파이프라인 무변경(회귀 0) +
서빙에서 손실됐던 polarity까지 복구. Option A(집계 변경=기존 keyword_match 회귀)·
C(병합) 기각, **B(sidecar) 채택**. → 결정 4 문구 개정 불요, 계획서 G1 데이터 소스만 명시.

그 외 3자 지적(G2 graph_view.js 실제 tooltip 작업·이웃 union 대칭·G4 explainer
_EDGE_MAP+provenance·shared_axes 라벨 sidecar·커버리지 하한 수치·gate-OFF IDF합·
G3 빈배열/404·demo attach 타이밍·§13.2 comparison 오기 정정)은 계획서/문서에 반영.
확정 결정과 무관한 보강이라 방향 변경 없음.
